# RFC v1.5 — AI Enrichment ("Wzbogać opis AI")

**Status:** DRAFT — wymaga decyzji user'a przed implementacją.
**Źródło:** port wzorca z mobile K16 (`project_inwentaryzacja_mobile.md`) + reuse logiki z `~/Projekty_software/muzeum-inwentarz/inwentarz.py`.
**Branch:** `feature/v1.5-ai-enrichment`.

## Problem

Desktop ma `description TEXT` w `eksponaty` ale opisy są **ręcznie wpisywane** przez usera w MainWindow. Często krótkie ("Amiga 1200"), bez kontekstu (era, charakterystyka, wartość historyczna).

Mobile workflow już ma AI identify (Claude Opus generuje pełny `Artefakt` ze zdjęcia). Desktop NIE — desktop edytuje istniejące rekordy z bazy bez AI.

**Cel:** w PreviewDialog (markdown render existing) dodać guzik "🪄 Wzbogać opis AI" który:
1. Pobierze zdjęcia eksponatu z bazy (BLOB photos)
2. Wyśle do Anthropic Claude z istniejącym `description` + meta (name/vendor/model) jako kontekst
3. Pokaże nowy opis w preview dialog (stary vs nowy markdown)
4. User akceptuje → UPDATE description w bazie

## Decision points (PROSZĘ POTWIERDZIĆ)

### 1. Architektura — A vs B vs C

**Opcja A: Direct C++ HTTP do Anthropic API** ⭐ rekomendowana
- Plus: standalone, brak Python dependency, zero dodatkowych deploy steps
- Plus: wzorce z mobile `ApiClient.cpp` (prepareRequest, formatNetworkError)
- Minus: ~250 LoC nowy kod (HTTP + JSON parsing + Anthropic schema duplicate w C++)

**Opcja B: C++ → wywołuje `inwentarz.py` przez QProcess**
- Plus: reuse istniejącego Python pipeline 1:1
- Minus: Python jako dependency desktopa (Windows installer = embedded Python), pip install, venv setup
- Minus: cross-platform deploy nightmare (.deb OK, .dmg + .exe trudne)

**Opcja C: C++ → REST do mobile FastAPI backend**
- Plus: 1 source of truth, thin client w desktop
- Minus: wymaga K14 deploy backendu na hosting (pending)
- Minus: dependency desktop działania na backend availability

### 2. API key storage

- `ANTHROPIC_API_KEY` env variable (devs OK, user awkward)
- QSettings (plaintext registry/ini — security trade-off, ale dla 1-user desktop OK)
- **Hybrid:** env var wins, fallback QSettings z prompt + ostrzeżeniem o lokalnym storage

### 3. UX confirm-dialog

- Ostrzeżenie kosztu przed call'em ("wyślę X zdjęć do AI, koszt ~$0.05, kontynuować?")
- Albo bezpośrednio call po kliknięciu (faster UX, ryzyko szybkiego $$)

### 4. Model

- **Claude Opus 4.7** (best quality, ~$5/1M in / $25/1M out, vision OK)
- Claude Sonnet 4.6 (3× tańszy, dobry dla większości eksponatów)
- User-configurable w Settings?

### 5. Input strategy

- **Refine** — wyślij existing description + meta + photos, prompt "wzbogać"
- **Replace** — zignoruj existing, generuj from scratch
- User wybiera per-call?

### 6. Save action

- Auto-update DB po Accept w preview dialog
- Albo preview only — user wraca do MainWindow do manual save (większa kontrola)

## Proponowana architektura (assuming A + hybrid key + Opus + refine + auto-save po confirm)

### Nowa klasa `AiEnrichmentService` (analogia do `DatabaseBackupService`)

```cpp
class AiEnrichmentService : public QObject {
    Q_OBJECT
public:
    explicit AiEnrichmentService(QObject *parent = nullptr);

    /// Async — emit `enrichmentReady(newDescription)` lub `enrichmentError(msg)`.
    void enrichDescription(const QString &existingDescription,
                          const QList<QByteArray> &photoBlobs,
                          const QString &itemName,
                          const QString &itemVendor,
                          const QString &itemModel);

signals:
    void enrichmentReady(const QString &newDescription);
    void enrichmentError(const QString &message);

private:
    QString readApiKey(QString *errorMessage) const;  // env var → QSettings → prompt
    QNetworkAccessManager *m_nam;
};
```

### Modyfikacje PreviewDialog

- Nowy guzik "🪄 Wzbogać opis AI" w buttonbox (obok "Edytuj")
- Click → fetch photos z bazy (BLOB SELECT), instantiate AiEnrichmentService
- Show modal `EnrichPreviewDialog` z BusyIndicator + status "Pytam AI..."
- Po `enrichmentReady` → side-by-side QTextBrowser (stary markdown | nowy markdown)
- Buttons: "Zachowaj stary" / "Zapisz nowy"
- Accept → ItemRepository.updateDescription(id, new) → reload PreviewDialog

### Nowa metoda w `ItemRepository`

```cpp
bool updateDescription(const QString &itemId, const QString &newDescription, QString *errorMessage);
```

## Pre-mortem (co może pójść nie tak)

- **Cost runaway:** user klika guzik 100× zamiast 5× → $5 spalony. Mitigation: confirm dialog z kosztem.
- **API key leak:** user zapomni że klucz w QSettings (plaintext). Mitigation: ostrzeżenie + opcja env var.
- **Schema drift:** Anthropic API zmienia format response → break. Mitigation: Pydantic-style schema validation w C++ (manual JSON checks).
- **Rate limit:** user batch processuje wiele eksponatów → 429. Mitigation: retry-after header support.
- **Model availability:** Opus 4.7 wycofany w przyszłości → fallback na Sonnet.
- **Photo BLOB size:** 200MB worth of photos in 1 record → request body za duży. Mitigation: limit max 5 photos lub resize do 1024px przed upload.
- **Markdown injection:** AI generuje markdown z linkami które otwiera QTextBrowser → potencjalny XSS desktop. Mitigation: `setOpenExternalLinks(false)` (już jest) + sanitize markdown przed render.

## Co NIE w MVP

- Cache odpowiedzi by-record (re-call koszt $$)
- Side-by-side diff visualizer (na razie 2 panes obok siebie wystarczy)
- Multi-eksponat batch enrichment (na razie 1 click = 1 record)
- A/B test różnych prompt wariantów
